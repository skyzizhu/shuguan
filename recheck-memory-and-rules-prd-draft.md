# 复审记忆系统与复审规则 PRD 草案

## 1. 文档目的

本文档用于定义内容审核产品中“复审”能力的后台记忆方案、复审触发规则和资源控制规则，作为后续产品设计、后端实现和模型编排的基础草案。

适用范围：

- 首轮审核已经完成的文字、文稿、图片内容
- 用户对首轮审核结果不满意、认为不准确，或希望进一步解释与纠偏
- 用户通过多轮对话或魔法棒动作发起复审，并在确认后更新结果页

## 2. 背景说明

当前系统支持三类审核物料：

1. 文字
2. 文件
3. 图片

现有 1.0.1 / 1.0.2 demo 中，首轮审核由大模型完成，并输出：

- 审校结论
- 政治敏感与合规性
- 内容质量与事实准度
- 法律与社会风气
- 舆情风险研判

复审不是重新执行一遍初审，而是在首轮审核结果基础上，针对用户质疑点或补充要求进行纠偏和再判断。

## 3. 复审的核心目标

复审阶段重点解决以下问题：

1. 首轮审核是否存在误判
2. 首轮审核是否存在漏判
3. 首轮解释是否不充分
4. 高风险内容的具体位置在哪里
5. 对应了哪些规则、平台规范或法律依据
6. 在补充上下文后，首轮结论是否需要调整

## 4. 是否需要记忆系统

### 4.1 结论

需要，但建议先做“审核会话记忆”，而不是通用长期记忆系统。

原因：

- 复审是围绕同一份内容的短周期纠偏任务
- 重点是保留“原始内容 + 首轮结果 + 多轮追问 + 复审意图”
- 不需要一开始就建设跨会话、跨任务的长期用户记忆

### 4.2 建议的最小可用记忆方案

后台建议存储以下几类信息：

1. 原始审核物料
   - 文字原文
   - 文件解析文本
   - OCR 结果
   - 图片视觉识别结果
   - 文件元数据

2. 首轮审核结果
   - 审校结论
   - 各维度结论
   - 命中的规则
   - 高风险片段
   - 解释依据

3. 复审会话消息
   - 用户每轮提问
   - 系统每轮回复
   - 魔法棒触发动作

4. 复审意图
   - 用户到底希望系统重新判断什么
   - 复审聚焦哪个维度
   - 复审范围是全文、某段还是某个区域
   - 用户认为首轮问题是什么

### 4.3 推荐的数据结构

建议先拆成以下几个对象：

- `review_session`
- `review_material`
- `initial_review_result`
- `review_followup_messages`
- `review_recheck_intent`

示例：

```json
{
  "session_id": "rs_001",
  "material_type": "document",
  "focus_dimensions": ["legal", "quality"],
  "target_scope": "全文第3-5段",
  "user_claims": [
    "认为首轮法律风险判断过严",
    "希望解释具体依据",
    "要求定位高风险段落"
  ],
  "expected_output": [
    "重新判断",
    "解释依据",
    "定位原句"
  ]
}
```

### 4.4 模型调用建议

点击“更新复审结果”时，不建议把整段聊天原样发给模型。

建议输入结构为：

- 原始物料内容
- 首轮审核结果
- 复审意图结构化结果
- 最近 2 到 4 轮关键对话摘要

这样可以：

- 降低 token 消耗
- 避免上下文污染
- 提高复审结果稳定性

## 5. 三类物料下的复审魔法棒

### 5.1 文字类

推荐按钮：

1. 复核疑似违规内容
2. 定位高风险段落
3. 解释判定依据
4. 排查误判内容

适用场景：

- 用户直接输入文案
- 用户质疑某段是否真的违规
- 用户希望知道依据或误判点

### 5.2 文件类

推荐按钮：

1. 复核疑似违规内容
2. 定位高风险段落
3. 解释规则与法律依据
4. 排查误判或漏判

适用场景：

- PDF / Word 文稿复审
- 用户希望定位到段落与原句
- 用户要求说明规则、法律、平台依据

### 5.3 图片类

推荐按钮：

1. 复核图片文字风险
2. 复核图片内容风险
3. 定位高风险区域
4. 解释判定依据

适用场景：

- OCR 文本已在首轮审核中完成识别
- 用户希望复核图片文字是否被误判
- 用户希望复核图片内容本身是否存在涉黄、涉毒或其他违规

## 6. 复审按钮激活规则

### 6.1 问题背景

“更新复审结果”按钮不能在用户刚说一句话后立即激活，否则会导致：

- 复审目标不明确
- 资源浪费
- 生成结果不稳定
- 用户体验不专业

### 6.2 核心原则

是否允许真正发起复审，不应只看“对话轮数”，而应看“复审意图是否成型”。

### 6.3 推荐激活条件

以下 3 类信息中，至少满足 2 类，按钮才可激活：

1. 复审对象明确
   - 例如：全文、某个维度、某一段、图片文字、图片区域

2. 复审目标明确
   - 例如：重新判断、解释依据、定位风险、排查误判、排查漏判

3. 复审理由明确
   - 例如：首轮判断过严、解释不充分、怀疑误判、怀疑漏判、需要结合上下文重审

### 6.4 可以激活的示例

示例 1：

“我觉得这段不应该算违规，请结合上下文重新审核。”

说明：

- 对象明确：这段
- 目标明确：重新审核
- 理由明确：不应该算违规

示例 2：

“请定位文稿里法律风险最高的原句，并重新判断是否过严。”

说明：

- 对象明确：文稿、原句
- 目标明确：定位 + 重新判断
- 理由明确：怀疑过严

### 6.5 不应激活的示例

示例 1：

“再看看。”

问题：

- 目标不明确
- 对象不明确

示例 2：

“为什么？”

问题：

- 只有解释倾向，但复审意图不完整

示例 3：

“我不认可这个结果。”

问题：

- 有情绪表达，但没有明确复审范围和目标

### 6.6 推荐后台返回结构

```json
{
  "ready_for_rereview": true,
  "missing_slots": [],
  "intent": {
    "scope": "内容质量与事实准度",
    "goal": "重新判断并定位高风险段落",
    "reason": "用户认为首轮判断过严"
  }
}
```

若条件不足，可返回：

```json
{
  "ready_for_rereview": false,
  "missing_slots": ["goal", "scope"]
}
```

前端可据此：

- 按钮置灰
- 给出提示文案
- 引导用户补充更明确的问题

## 7. 资源控制建议

为避免复审被滥用，建议增加以下资源保护策略：

1. 每次点击“更新复审结果”前，必须满足复审意图成型条件
2. 同一会话内连续复审次数可设上限
3. 若用户未产生新的有效复审意图，不重复调用模型
4. 复审优先基于结构化意图，不直接吃全量聊天记录
5. 可对高成本图片复审设置更严格的激活规则

## 8. 当前 demo 的落地方向

当前 demo 已体现以下交互方向：

- 复审魔法棒会根据物料类型动态切换
- 用户可通过多轮对话继续纠偏
- 对话后不会立即替换结果页
- 只有点击“更新复审结果”后，第二屏结果才统一更新

当前 demo 仍属于演示版，后续真正上线时，建议再补：

- 后台复审意图识别
- 复审任务状态流转
- 复审结果版本管理
- 首轮结果与复审结果的差异对比

## 9. 后续建议

下一步推荐继续补 3 个产品点：

1. 复审结果与首轮结果的差异展示方式
2. 复审按钮激活状态的前端提示语
3. 复审完成后是否保留“首轮结果 / 复审结果”双版本视图
